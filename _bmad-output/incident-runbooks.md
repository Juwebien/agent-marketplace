## Scénario 1: EXPLOIT SMART CONTRACT (Drain Escrow)

### Détection
- Configurer alertes Grafana : Monitorer solde escrow via API Base L2 (seuil : chute >10% en 5min). Alertes sur métriques de tx inhabituelles (spikes >500% volume normal).
- Alertes Telegram : Bot notifiant #alerts-critiques avec détails tx (hash, montant drainé) via webhook de nœud Base.

### T+0 à T+30min: Actions Immédiates
- **DevOps Lead** : Vérifier dashboards Grafana pour confirmer exploit (5min). Stopper tous les pods k3s liés à l'indexer et API frontend via `kubectl scale deployment frontend --replicas=0`.
- **Security Engineer** : Scanner tx sur Base explorer (e.g., basescan.org) pour identifier adresse attaquante. Exécuter script d'urgence pour pauser contrat escrow via multisig (clé privée sécurisée dans vault).
- **Smart Contract Dev** : Confirmer vuln via code audit rapide (10min). Si confirmé, déclencher fonction pause() si disponible.
- **CEO/Founder** : Approuver pause contrat (via multisig vote si needed).
- Checklist globale : [ ] Confirmer exploit [ ] Pauser contrats [ ] Notifier équipe via Telegram #crisis.

### T+1h à T+24h: Stabilisation
- **Security Engineer** : Déployer contrat patché sur testnet Base, tester drain-proof. Migrer fonds restants vers nouveau escrow sécurisé.
- **DevOps Lead** : Restaurer indexer DB à partir backup (k3s volume snapshot, <1h). Redémarrer pods avec monitoring renforcé.
- **Smart Contract Dev** : Auditer full code pour autres vulns, déployer upgrade sur mainnet via governance proposal.
- **Toute équipe** : Monitorer pour follow-up attacks (Grafana alerts x2 sensibilité).

### Communication
- **Providers** : Email direct "Pause temporaire des contrats due à incident de sécurité. Fonds en migration. Pas de pertes confirmées pour vous."
- **Clients** : Notification in-app + email "Service pausé pour maintenance urgente. Mises à jour dans 1h."
- **Community Discord/Twitter** : Post immédiat Discord #annonces : "Détection d'exploit potentiel sur escrow. Contrats pausés. Équipe en action. Pas de panique." Twitter : "🚨 Incident de sécurité détecté. Services pausés. Mises à jour à venir. #Web3Marketplace"

### Post-mortem Requis
- Réunir équipe 72h max. Rapport : Analyse tx exploit, timeline réponse, updates à Grafana alerts.

## Scénario 2: INSURANCE POOL DRAINÉ (Attaque Sybil Disputes)

### Détection
- Configurer alertes Grafana : Monitorer solde pool via smart contract queries (seuil : drain >20% en 1h). Détecter spikes en disputes ( >300% normal).
- Alertes Telegram : Bot notifiant #alerts-critiques avec détails (nombre disputes, adresses suspectes) via subgraph queries sur Base.

### T+0 à T+30min: Actions Immédiates
- **Security Engineer** : Analyser disputes via subgraph (The Graph) pour patterns sybil (mêmes IPs/adresses liées). Bloquer adresses suspectes via contrat admin fonction.
- **Smart Contract Dev** : Pauser fonction dispute() et withdrawals du pool via multisig.
- **DevOps Lead** : Vérifier logs k3s pour anomalies (e.g., `kubectl logs -n marketplace dispute-processor`).
- **CEO/Founder** : Approuver pause et notifier assureurs partenaires.
- Checklist globale : [ ] Identifier sybil patterns [ ] Pauser disputes [ ] Sauvegarder pool état actuel.

### T+1h à T+24h: Stabilisation
- **Security Engineer** : Implémenter CAPTCHA/KYC temp sur disputes pour bloquer sybil. Rembourser disputes légitimes manuellement.
- **Smart Contract Dev** : Déployer upgrade contrat avec anti-sybil (e.g., proof-of-uniqueness via ZK).
- **DevOps Lead** : Augmenter replicas dispute-processor pods à 3 pour résilience, sync DB.
- **Toute équipe** : Scanner pour pools secondaires affectés, migrer fonds.

### Communication
- **Providers** : Email "Pool insurance drainé par disputes suspectes. Pause en cours. Remboursements prioritaires pour vous."
- **Clients** : Notification in-app "Disputes pausées due à activité anormale. Vos fonds sécurisés."
- **Community Discord/Twitter** : Discord #annonces : "Attaque sybil détectée sur pool. Pausé. Équipe résout." Twitter : "🛡️ Attaque sur insurance pool. Services limités. Mises à jour soon. #AIWeb3"

### Post-mortem Requis
- Réunir équipe 72h max. Rapport : Analyse sybil methods, efficacité pause, nouvelles règles anti-sybil.

## Scénario 3: INDEXER DOWN 48H (DB Désync)

### Détection
- Configurer alertes Grafana : Monitorer uptime indexer (seuil : downtime >5min) et sync lag DB (>1 block Base L2).
- Alertes Telegram : Bot notifiant #alerts-critiques avec détails (erreur logs, lag en blocks) via k3s healthchecks.

### T+0 à T+30min: Actions Immédiates
- **DevOps Lead** : Vérifier status k3s `kubectl get pods -n indexer`, restart pods down via `kubectl rollout restart deployment indexer`.
- **Security Engineer** : Scanner pour attaques (e.g., DDoS via logs Cloudflare/Base node).
- **Smart Contract Dev** : Vérifier subgraph sync sur The Graph, resync manuellement si needed.
- **CEO/Founder** : Approuver bascule vers backup indexer si dispo.
- Checklist globale : [ ] Diagnostiquer cause [ ] Restart pods [ ] Vérifier sync.

### T+1h à T+24h: Stabilisation
- **DevOps Lead** : Restaurer DB à partir snapshot (k3s persistent volume, <2h). Augmenter resources CPU/RAM pods à 2x.
- **Security Engineer** : Ajouter rate limiting sur API indexer pour prévenir overload.
- **Smart Contract Dev** : Resync full DB avec Base L2 node (script custom, estim 4-6h).
- **Toute équipe** : Tester frontend post-resync, monitorer pour re-down.

### Communication
- **Providers** : Email "Indexer down, queries en retard. Restauration en cours. Pas d'impact sur fonds."
- **Clients** : Notification in-app "Délais sur mises à jour marché due à maintenance. Back soon."
- **Community Discord/Twitter** : Discord #status : "Indexer offline >48h. Équipe sur le fix. Pas de pertes." Twitter : "⚠️ Indexer down. Stabilisation en cours. #BaseL2Marketplace"

### Post-mortem Requis
- Réunir équipe 72h max. Rapport : Cause downtime, temps restauration, améliorations HA (e.g., multi-node k3s).
