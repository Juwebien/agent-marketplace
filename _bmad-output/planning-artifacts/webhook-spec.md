# Spécification Technique des Webhooks pour Marketplace AI Agent sur Base L2

Cette spécification définit les webhooks pour un marketplace d'agents AI sur la blockchain Base Layer 2 (L2). Les webhooks sont implémentés avec Fastify v5 en TypeScript. Ils notifient les clients (e.g., frontends, intégrations tierces) des changements d'état des missions et des matchs avec des providers.

Les webhooks sont envoyés via HTTP POST vers une URL configurée par le client. Chaque webhook inclut un payload JSON, une signature HMAC-SHA256 pour la vérification, et supporte les retries avec exponential backoff. L'idempotence est gérée côté client via un `event_id` unique.

## 1. Types d'Événements

Les webhooks supportent les types d'événements suivants, basés sur les états de mission et les matchs de providers :

- **MISSION_CREATED**: Une nouvelle mission est créée.
- **MISSION_FUNDED**: La mission est financée (e.g., fonds escrow sur Base L2).
- **MISSION_ACCEPTED**: Un provider accepte la mission.
- **MISSION_IN_PROGRESS**: La mission est en cours d'exécution.
- **MISSION_DELIVERED**: Le provider livre le résultat.
- **MISSION_COMPLETED**: La mission est complétée et validée.
- **MISSION_DISPUTED**: Un litige est ouvert.
- **MISSION_REFUNDED**: Les fonds sont remboursés.
- **MISSION_CANCELLED**: La mission est annulée.
- **PROVIDER_MATCHED**: Un provider est matché avec la mission (e.g., via algorithme AI).

Chaque événement est déclenché par des smart contracts sur Base L2 ou des services backend, et dispatché via le serveur Fastify.

## 2. Format JSON des Payloads

Tous les payloads sont au format JSON avec les champs suivants (communs à tous les événements) :

- `event_type`: String (un des types ci-dessus).
- `event_id`: String (UUID unique pour l'idempotence).
- `timestamp`: Number (timestamp Unix en millisecondes).
- `mission_id`: String (ID unique de la mission sur Base L2).
- `data`: Object (données spécifiques à l'événement).

## 3. HMAC-SHA256 Signature Headers

Pour sécuriser les webhooks, chaque requête inclut un header `X-Signature` calculé via HMAC-SHA256.

- **Clé Secrète**: Une clé partagée (générée par client, stockée securely côté serveur).
- **Message à Signer**: Concaténation de `event_id` + `timestamp` + JSON.stringify(payload.data) (sans espaces).
- **Calcul**: `crypto.createHmac('sha256', secret).update(message).digest('hex')`.
- **Header**: `X-Signature: hmac-sha256=<signature-hex>`.
- **Vérification Côté Client**: Recalculer la signature et comparer.

Exemple de message à signer : `"uuid-1234-56781698765432000{"creator_address":"0x1234567890abcdef","description":"Développer un agent AI pour analyse de données","budget":"100 USDC"}"`.

## 4. Retry Logic avec Exponential Backoff

Le dispatcher réessaie les envois échoués (e.g., non-200 HTTP) avec exponential backoff :
- Max retries : 5.
- Délai initial : 1 seconde.
- Facteur backoff : 2 (délai = initial * (2 ^ attempt)).
- Jitter : Ajouter un random de 0-500ms pour éviter les collisions.

Exemple : Retry 1: 1s + jitter; Retry 2: 2s + jitter; Retry 3: 4s + jitter; etc.

Si tous les retries échouent, logger l'erreur et notifier admin (non inclus dans le code ci-dessous).

## 5. Code TypeScript du Dispatcher

Voici le code TypeScript pour le dispatcher Fastify. Il suppose un setup Fastify existant et utilise `node-fetch` pour les requêtes HTTP et `crypto` pour HMAC. Intégrez-le dans votre serveur.

```typescript
import fastify from 'fastify';
import fetch from 'node-fetch';
import { randomUUID } from 'crypto';
import { createHmac } from 'crypto';

// Interface pour le payload
interface WebhookPayload {
  event_type: string;
  event_id: string;
  timestamp: number;
  mission_id: string;
  data: Record<string, any>;
}

// Fonction pour signer le payload
function signPayload(payload: WebhookPayload, secret: string): string {
  const message = `${payload.event_id}${payload.timestamp}${JSON.stringify(payload.data)}`;
  return createHmac('sha256', secret).update(message).digest('hex');
}

// Fonction de dispatch avec retries
async function dispatchWebhook(url: string, payload: WebhookPayload, secret: string, attempt = 1): Promise<void> {
  const signature = signPayload(payload, secret);
  const response = await fetch(url, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Signature': `hmac-sha256=${signature}`,
    },
    body: JSON.stringify(payload),
  });

  if (response.ok) {
    console.log(`Webhook sent successfully: ${payload.event_id}`);
    return;
  }

  if (attempt >= 5) {
    console.error(`Failed to send webhook after 5 attempts: ${payload.event_id}`);
    return;
  }

  const delay = 1000 * Math.pow(2, attempt) + Math.random() * 500; // Exponential backoff + jitter
  console.log(`Retrying webhook ${payload.event_id} in ${delay}ms (attempt ${attempt + 1})`);
  setTimeout(() => dispatchWebhook(url, payload, secret, attempt + 1), delay);
}

// Exemple d'intégration dans Fastify (appelée quand un événement se produit)
const server = fastify();

server.post('/trigger-webhook', async (request, reply) => {
  // Exemple: Déclenché par un événement (remplacez par votre logique)
  const payload: WebhookPayload = {
    event_type: 'MISSION_CREATED',
    event_id: randomUUID(),
    timestamp: Date.now(),
    mission_id: 'mission-abc123',
    data: { /* ... */ },
  };
  const clientUrl = 'https://client-webhook-url.com'; // Récupéré de DB
  const clientSecret = 'secret-key-from-db'; // Récupéré de DB

  dispatchWebhook(clientUrl, payload, clientSecret);
  return { status: 'Webhook dispatched' };
});

// Démarrer le serveur (exemple)
server.listen({ port: 3000 }, (err) => {
  if (err) throw err;
  console.log('Server listening on port 3000');
});
```

## 6. Idempotency Handling Côté Client

Pour éviter les duplications (e.g., dues à retries), le client doit implémenter l'idempotence :
- Utiliser `event_id` comme clé unique (stockée en DB ou cache, e.g., Redis avec TTL 24h).
- À réception :
  1. Vérifier si `event_id` existe déjà → Si oui, répondre 200 sans traitement.
  2. Vérifier la signature HMAC.
  3. Traiter l'événement.
  4. Stocker `event_id` comme traité.
- Répondre toujours avec HTTP 200 pour succès (même si idempotent), ou 4xx/5xx pour erreurs (déclenche retry).
